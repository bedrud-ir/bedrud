# Bedrud Mobile Feature Plan
## Web → Android & iOS parity

Generated by analysis of `apps/web/src` on 2026-04-04.
Each section notes what the web has, what status mobile has, and what needs building.

---

## Legend
- ✅ Already implemented in both platforms
- 🤖 Android only
- 🍎 iOS only  
- ❌ Missing from both
- ⚠️ Partially implemented

## 1. Authentication

### 1.1 Email / Password Login
- ✅ Both platforms have this
- Web extras to match:
  - Show / hide password toggle button
  - Per-field inline validation (not just toast errors)
  - Global error banner (styled, not just a Snackbar)

### 1.2 Passkey Login (WebAuthn / FIDO2)
- ✅ Both platforms have PasskeyManager

### 1.3 OAuth Login — Google, GitHub, Twitter/X
- ❌ Missing from both
- Web hits `GET /api/auth/{provider}/login` which redirects to OAuth provider
- Mobile should open a Custom Tab (Android) / ASWebAuthenticationSession (iOS)
- Handle the OAuth callback redirect back into the app
- Providers: `google`, `github`, `twitter`

### 1.4 Guest Mode (no account required)
- ❌ Missing from both
- API: `POST /api/auth/guest-login { name }` → returns full AuthResponse
- User enters a display name (≥ 2 chars), receives a temporary token
- Guest users land on Dashboard just like registered users
- Web shows this as a separate "Join as guest" page accessible from Login/Register

### 1.5 Registration
- ✅ Both platforms have this
- Web extras to match:
  - Confirm-password field
  - Invite token field (conditionally shown when server requires it — fetch `/api/auth/settings`)
  - `tokenRegistrationOnly` flag from public settings hides/shows token field

### 1.6 Auth Token Persistence
- ✅ Both platforms handle tokens
- Web has a "remember me" toggle (localStorage vs sessionStorage)
- Mobile equivalent: persist to Keychain/EncryptedSharedPrefs by default; optionally clear on app close

## 2. Dashboard

### 2.1 Room List with Stats Header
- ⚠️ Basic room list exists in both
- Missing: three stat chips at top — **Total**, **Live now** (active), **Private**
- Counts come from the room list itself (filter client-side)

### 2.2 Filter Tabs (All / Active / Private)
- ❌ Missing from both
- Segment control (iOS) / FilterChip row (Android) below stats
- Filters the room list in-place

### 2.3 Quick Join Bar
- ❌ Missing from both
- Prominent search/input field: "Enter a room name or code to join instantly"
- Normalizes input (lowercase, spaces → dashes) then navigates to `/m/{roomSlug}`

### 2.4 Create Room Dialog — Full Feature Set
- ⚠️ Android has RoomSettingsDialog; iOS has basic create
- Web's create dialog fields:
  - **Room name** (optional; auto-generated if blank)
  - **Max participants** (number input, 2–500, default 20)
  - **Visibility** — Private / Public (two-option selector)
  - **Feature toggles** (each has icon + label + description):
    - Audio (allow participants to speak)
    - Video (allow participants to share camera)
    - Chat (in-room text chat)
    - End-to-end encryption (E2EE)
  - On submit: `POST /api/room/create` → navigate directly into the new room

### 2.5 Room Card
- ⚠️ Both have basic cards
- Web shows:
  - Room name
  - Active / Idle badge (green pulse when active)
  - Public / Private badge
  - Feature pills (Chat, Video, E2EE) when enabled
  - Max participants
  - **Join** button → navigates to meeting
  - **Delete** button (with error feedback)

### 2.6 Delete Room
- ❌ Missing from both
- `DELETE /api/room/{id}`
- Show error banner if it fails (auto-dismiss after 4 s)

### 2.7 Greeting
- ❌ Minor — web shows "Good morning/afternoon/evening, {firstName}'s rooms"
- Nice to have, not critical

## 3. Meeting Room

### 3.1 Join Flow
- ✅ Authenticated join (`POST /api/room/join { roomName }`) — both platforms
- ❌ **Guest join** — `POST /api/room/guest-join { roomName, guestName }`
  - If the user is not authenticated, show a "Join as guest" sheet/dialog before connecting
  - User picks a display name, then the API call is made
- ✅ Loading spinner while waiting for join response
- ❌ **Join error screen** — currently unclear if mobile handles the error state with a dedicated UI

### 3.2 Meeting Header (HUD)
- ❌ Missing from both (iOS has basic ControlBar but no header)
- Fixed overlay at the top of the meeting screen
- Shows: LIVE badge, room name (monospace), participant count, current time, connection state dot (green=Connected, yellow=Connecting)

### 3.3 Controls Bar
- ⚠️ iOS has ControlBar; Android has MeetingScreen controls
- Web controls (all needed):
  - **Mic toggle** — with device selector caret (pick input device)
  - **Camera toggle** — with device selector caret
  - **Leave / End button** (red, prominent)
  - **Speaker output selector** caret
  - **Chat toggle** — with unread count badge
  - **Participants toggle**

### 3.4 Push-to-Talk / Push-to-Mute
- ❌ Missing from both
- Web: hold Space bar to temporarily toggle mic
- Mobile: long-press a dedicated PTT button in controls
- Show a floating badge ("Push-to-Talk active" / "Push-to-Mute active") while held

### 3.5 Device Selector (mic / camera / speaker)
- ❌ Missing from both
- Web: small caret next to each device button opens a dropdown to pick device
- Mobile: tap caret → bottom sheet with available devices listed
- Uses LiveKit's `getLocalDevices()` equivalent

### 3.6 Chat Panel
- ❌ Missing from both
- Slide-in side panel (Android: slide from right edge; iOS: sheet or overlay)
- Real-time messages via LiveKit data channel (LiveKit `useChat` equivalent)
- Always listening even when panel is closed (accumulates unread count)
- Features:
  - Messages list: own messages right-aligned (indigo bubble), others left-aligned (dark bubble)
  - Sender name shown above others' messages
  - System event rows (kick/ban notifications) — centered italic pill
  - Auto-scroll to bottom on new message
  - Text input + send button
  - Empty state ("No messages yet. Say hello!")
  - Mark-read when panel opens (clears unread count)

### 3.7 Unread Chat Badge
- ❌ Missing from both
- Red dot / number badge on the chat toggle button
- Counts chat messages + system messages received while panel is closed
- Clears when panel is opened

### 3.8 Participants Panel
- ❌ Missing from both
- Slide-in panel (same side as chat, mutually exclusive)
- Shows all connected participants
- Per participant:
  - Gradient avatar (deterministic color from name initial)
  - Display name + "you" label if local
  - Role badges: **Admin**, **Mod**, **Guest**
  - Mic status icon (gray mic = on, red mic-off = muted)
  - Camera status icon
- **Moderation actions** (visible to room admin / moderators only):
  - Three-dot menu per participant (not self)
  - **Mute Audio** — `POST /api/room/{roomId}/mute/{identity}`
  - **Kick** — `POST /api/room/{roomId}/kick/{identity}`
  - **Ban** (admin only) — `POST /api/room/{roomId}/ban/{identity}`

### 3.9 System Event Messages
- ❌ Missing from both
- LiveKit data messages on topic `"system"` carry kick/ban events
- Parse `{ type: "system", event: "kick"|"ban", actor, target }` 
- Inject as inline system rows into the chat timeline
- Format: `"{target} was kicked/banned by {actor}"`

### 3.10 Kick Detection
- ❌ Missing from both (unknown if Android handles it)
- Listen for LiveKit `DisconnectReason.PARTICIPANT_REMOVED`
- Show a dedicated "You were removed" screen (not just a toast)
- Button to go back to Dashboard

### 3.11 Leave / End Meeting Dialog (for room creator)
- ⚠️ Unknown state
- If current user is the room creator (`currentUserId === adminId`):
  - Tapping Leave shows a dialog with three options:
    1. **End Meeting for Everyone** — `DELETE /api/room/{roomId}` then disconnect
    2. **Leave Meeting** — just disconnect (room stays alive)
    3. **Cancel**
- If not creator: leave immediately on button tap

### 3.12 Participant Grid (video tiles)
- ✅ Both platforms have RoomManager with LiveKit; MeetingScreen exists
- Ensure the grid adapts: 1 participant (spotlight), 2–4 (2-col grid), 5+ (scrollable grid)

## 4. User Settings

### 4.1 Profile — Update Display Name
- ✅ Both have ProfileScreen/ProfileView
- API: `PUT /api/auth/me { name }`
- Show success/error inline alert after save

### 4.2 Change Password
- ❌ Missing from both
- Fields: current password, new password, confirm password
- `PUT /api/auth/password { currentPassword, newPassword }`
- Validation: new ≥ 6 chars, must match confirm
- If user signed in via OAuth: show a message instead of the form ("Password managed by {provider}")

### 4.3 Account Info (read-only)
- ❌ Missing from both
- Rows: Account ID, Sign-in method, Role (User / Superadmin)
- If superadmin: show a "Superadmin — full system access" badge

## 5. Admin Panel

> The entire admin panel is missing from both mobile apps.
> Superadmin users should see an "Admin" tab or section accessible from the dashboard.

### 5.1 System Overview
- Stats grid (6 cards): Total users, Total rooms, Online users, Admins, Public rooms, Active rooms
- Server health status indicator (green pulse)
- Room creation activity chart — last 7 days (bar or area chart)
- Recent sign-ups list (last 5 users, with provider badge)
- Room breakdown bar chart (live / public / private ratios)

### 5.2 Users Management
- Full user list from `GET /api/admin/users`
- Search by name or email
- Sort by name, email, provider, join date
- Per user row:
  - Name, email, provider badge (local/google/github/guest/passkey)
  - Active / Banned status toggle — `PUT /api/admin/users/{id}/status { active }`
  - Promote / Demote superadmin — `PUT /api/admin/users/{id}/accesses { accesses }`
- Tap user → user detail screen (basic info + actions)

### 5.3 Rooms Management
- Full room list from `GET /api/admin/rooms`
- Search by name
- Sort by name, creation date, max participants
- Per room row:
  - Room name, visibility badge, active/idle badge
  - Participant limit (tap to edit inline)
  - Force-close active room — `DELETE /api/admin/rooms/{id}` (with confirmation)

### 5.4 System Settings
- Registration settings from `GET /api/admin/settings`
- Toggle: **Allow new registrations** — `PUT /api/admin/settings { registrationEnabled }`
- Toggle: **Require invite token** — `PUT /api/admin/settings { tokenRegistrationOnly }`
- Invite tokens section:
  - List from `GET /api/admin/invite-tokens`
  - Per token: truncated token string, optional locked email, expiry date, status badge (Valid/Used/Expired)
  - Copy to clipboard button
  - Delete / revoke — `DELETE /api/admin/invite-tokens/{id}`
  - Generate new token form:
    - Optional email lock field
    - Expiry selector: 24h / 72h / 7 days / 30 days
    - `POST /api/admin/invite-tokens { email, expiresInHours }`
    - Show generated token immediately with copy button
